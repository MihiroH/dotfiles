# アンチパターン集 & 提出前チェックリスト

ireko ファイルのレビュー・自己チェックに使う。どれも実在する失敗パターンに基づく。

## 構造的アンチパターン

### AP-1: 循環 ref (ビルドエラー)

**症状:** `A → B → A` の ref が存在する。ireko がエラーで止まる。

**よくある発生パターン:**
- 詳細図から共通ハブに「戻り ref」を張ってしまう
  ```
  // 共通ハブ
  diagram TLSInProxies "..." {
    ref Term > TLSTermination
  }
  // 詳細図
  diagram TLSTermination "..." {
    ref RP > TLSInProxies   // ← これが循環
  }
  ```

**対策:** ref は常に「親 → 子」の一方向。戻りはパンくずで辿れるので不要。

### AP-2: 孤立ダイアグラム (警告)

**症状:** どこからも ref されていないダイアグラムがある。root から到達できない。

**対策:** root から BFS で到達可能か確認。到達できないものは親から ref を張るか、不要なら削除。

### AP-3: 未定義 ref (ビルドエラー)

**症状:** `ref NonExistent` で存在しない識別子を参照。

**対策:** 定義前の typo が多い。`grep "^diagram" file.ireko` で全識別子を確認。

### AP-4: ルートに情報を詰め込みすぎ

**症状:** ルートダイアグラムに 10+ ノードと説明テキストが詰め込まれている。

**ダメな例:**
```
@root
diagram "全体像" {
  flowchart TB
    A[長い説明文がここに] --> B[さらに長い説明]
    B --> C[詳細な仕様の記述]
    ...
}
```

**対策:** ルートは「この先に何があるか」の目次。詳細は ref で追い出す。ルートは 1 画面に収まるサイズにする。

### AP-5: Why が浅い

**症状:** 「なぜ使うか」が flowchart 1 枚で終わり、各目的から How への ref が張られていない。

**対策:** Why ダイアグラムの各目的ノードから、それを実現する How 詳細図へ必ず ref する。これが無いと「読者は Why と How を別々に読む」ことになり、繋がりが見えない。

**良い例:**
```
diagram WhyUseIt "なぜ使うか" {
  flowchart TB
    Why([目的]) --> G1[負荷分散]
    Why --> G2[TLS 終端]

    ref G1 > LoadBalancing   // How への ref
    ref G2 > TLSTermination  // How への ref
}
```

### AP-6: 共通テーマの重複

**症状:** 同じトピック (例: X-Forwarded-For) が複数のダイアグラムで独立に説明されている。

**対策:** 共通ハブを 1 つ作り、各所からそこに ref する (ただしハブから各所への戻りは張らない)。

### AP-7: ネストが深すぎる

**症状:** ルート → カテゴリ → サブカテゴリ → 詳細 → 補足 → 実装 → ... と 5 階層以上。

**対策:** 3〜4 階層が目安。深くしたくなったら:
- 兄弟関係にする (横に広げる)
- 中間階層を削る
- そもそも別ファイルに分ける (プロジェクト規模が大きい場合)

## 図選択のアンチパターン

### AP-8: flowchart 偏重

**症状:** 全部のダイアグラムが flowchart。

**対策:** `reference/diagram-selection.md` に従って選び直す。特に:
- 比較 → classDiagram
- 時系列 → sequenceDiagram
- 状態遷移 → stateDiagram-v2

### AP-9: mindmap をルートに使う

**症状:** ルートが mindmap で、各枝が ref に単に並んでいる。

**問題:** 学習順序が見えない。放射状で「どこから読むか」が曖昧。

**対策:** flowchart TB に変換し、「ここから始める」の矢印で順路を明示する。

### AP-10: flowchart の単純な箇条書き化

**症状:** ノード → 説明テキストの 1:1 対応で、ツリーが展開されているだけ。

**ダメな例:**
```
flowchart TB
    A([目的]) --> B[理由 1 の説明テキスト]
    A --> C[理由 2 の説明テキスト]
    A --> D[理由 3 の説明テキスト]
```

**対策:**
- 親ノードに要点を統合し、説明文を外す
- または各理由を ref で詳細図に追い出す

### AP-11: sequenceDiagram の participant 過多

**症状:** 7 人以上の participant が登場し、左右にスクロールしないと読めない。

**対策:** 機能単位で分割し、「Client → Gateway」と「Gateway → Backend」を別図にする。

### AP-12: stateDiagram を非排他的な状態に使う

**症状:** stateDiagram なのに「A でもあり B でもある」状態を書いている。

**対策:** 排他的でないなら flowchart のほうが適切。stateDiagram は「ある時点ではどれか 1 つの状態」が大前提。

## 記述スタイルのアンチパターン

### AP-13: ノードラベルの日本語と記号のミックスで Mermaid がエラー

**症状:** `A[目的: 複数Backend に振り分け (負荷分散)]` のように、コロン・括弧・改行が混在。

**対策:**
- ノードラベルは `"..."` で囲む: `A["目的: 複数..."]`
- 改行は `<br/>` (閉じタグ必須)
- セミコロン `;` は避ける (Mermaid が誤解釈する)

### AP-14: classDiagram の属性に空白やコロンの記述ミス

**症状:** `+属性:値` のようにコロン後に空白がない、またはコロン前に空白がある。

**対策:** `+属性: 値` (コロン直後に半角空白) で統一。

### AP-15: 識別子の命名が一貫しない

**症状:** ある所では `TLSTermination`、別の所では `tlsTermination` や `TLS_Termination`。

**対策:** UpperCamelCase で統一。ireko の識別子は case-sensitive。

### AP-16: 識別子がトピックを表していない

**症状:** `Detail1`, `SubA`, `Fig3` のような意味の無い識別子。

**対策:** 識別子を見ただけで中身が分かる名前にする。`TLSHandshakeDetail`, `AuthFlowSequence`。

## 提出前チェックリスト

書き終わったら以下を上から順に確認する。

### 構造

- [ ] `@root` はちょうど 1 つか
- [ ] 循環 ref はないか (検証スクリプトで確認)
- [ ] 未定義 ref はないか (検証スクリプトで確認)
- [ ] root から全ダイアグラムに到達可能か
- [ ] ref は親 → 子の一方向になっているか (子 → 親の戻り ref はないか)

### ルートダイアグラム

- [ ] ルートは 1 画面に収まる大きさか
- [ ] ルートは「目次」として機能しているか (詳細は ref で追い出されているか)
- [ ] ルートの図の種類は flowchart か? (mindmap なら再検討)

### Why と How の接続

- [ ] Why ダイアグラムがあるなら、各目的から How 詳細へ ref しているか
- [ ] 共通トピックは 1 つのハブに集約されているか

### 図の種類

- [ ] 全部 flowchart になっていないか
- [ ] 比較系は classDiagram になっているか
- [ ] 時系列は sequenceDiagram になっているか
- [ ] 状態遷移は stateDiagram-v2 になっているか

### ネスト

- [ ] ネストは最大 4 階層までか
- [ ] 孤立した深い枝はないか

### 記述

- [ ] 識別子は UpperCamelCase で統一されているか
- [ ] 識別子はトピックを表しているか (Detail1 のような命名はないか)
- [ ] Mermaid のノードラベルに `;` が使われていないか
- [ ] 改行は `<br/>` か (`<br>` ではなく)
- [ ] classDiagram の属性は `+key: value` の形式か

### 内容

- [ ] 各ダイアグラムに単独で意味が通じるタイトルが付いているか
- [ ] ダイアグラム 1 つが扱うトピックは 1 つか (欲張って複数トピックを混ぜていないか)

## レビューの書き方 (他人のファイルを指摘する時)

レビューを書くときは以下の順で指摘を整理すると読み手が動きやすい:

1. **致命的問題** (ビルドエラー: 循環 ref, 未定義 ref, `@root` 不整合)
2. **構造的問題** (Why が浅い, ルートが重すぎる, 共通テーマの重複)
3. **図選択の問題** (flowchart 偏重など)
4. **記述スタイル** (命名, Mermaid 記法)
5. **追加すべきトピック** (内容面の欠落)

優先度の高いものから先に直す。1 と 2 が解決しないと他を直しても意味がないため。
